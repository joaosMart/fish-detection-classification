Three Approaches to Rebuild the Viewer

Choose the architecture direction — all fix the broken timeline clicks

A

Enhanced Static HTML (Recommended)

Keep the single-file HTML generator but completely redesign the UI. Add a video list sidebar with detection counts, a persistent video player that stays in view, and a unified timeline merging single + multi detections with color-coded markers. Fix the broken click handlers. Add keyboard shortcuts (N/P for next/prev detection).

Pros

  • No new dependencies — still just Python + generated HTML
  • Works offline, easy to share the HTML file
  • Fastest to implement

Cons

  • Still one big generated HTML file
  • Limited interactivity without a framework
B

Lightweight SPA (Vanilla JS + JSON)

Generate a data JSON file from Python, then serve a separate static HTML/CSS/JS app that reads it. The app has a proper layout: sidebar for video navigation, main area for player + timeline, and a mini dashboard header. Templates separated from data.

Pros

  • Clean separation of data and UI
  • Easier to iterate on UI without touching Python
  • Still no framework dependencies

Cons

  • More files to manage (HTML + JS + CSS + JSON)
  • Slightly more complex deployment
C

Flask/FastAPI Web App

Full Python web app with a proper backend. Real-time video streaming, database for results, REST API. Could add user annotations, threshold tuning UI, and session management.

Pros

  • Most powerful and extensible
  • Could add collaborative features

Cons

  • Significant new dependencies
  • Overkill for the current use case
  • Much longer to build
